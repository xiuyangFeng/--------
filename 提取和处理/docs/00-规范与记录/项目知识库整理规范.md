# 项目知识库整理规范

> 适用范围：本仓库的 README、AGENTS.md、docs、实验路线记录、外部 baseline 复现文档和推进记录。

本规范用于把项目管理从“不断追加记录”收敛为“入口清楚、路线分层、状态可追踪”的工程化知识库。核心原则是：事实写到最贴近事实的位置，入口只做导航，推进记录只做按时间倒序的变更凭证。

## 1. 文档分层

| 层级 | 主要位置 | 职责 |
| --- | --- | --- |
| 项目规则 | `AGENTS.md` | 环境、禁止事项、推进记录和提交规则 |
| 人类入口 | `README.md`、`docs/README.md` | 当前结构、快速导航、稳定运行入口 |
| 研究总纲 | `docs/实验设计总纲.md` | 论文主线、任务拆分、路线定位、阶段性假设 |
| 路线状态 | `docs/01-任务/任务A/**` | V1/V2/V3 的实验日志、状态表、待办和 Go/No-Go |
| 规范记录 | `docs/00-规范与记录/**` | 评估口径、记录模板、知识库整理规则 |
| 推进变更 | `docs/02-推进与变更/代码修改与实验推进记录.md` | 代码、配置、脚本和实验文档变更的倒序记录 |
| 论文复现 | `docs/paper_reproduction/**`、`external_baselines/**` | 外部 baseline、私有数据适配、后处理和可视化交付 |

## 2. 变更同步规则

- 训练、评估、loss、dataset、pipeline、Slurm 或配置发生变化时，先更新对应代码目录 README 或路线文档，再更新推进记录。
- 外部论文 baseline 不写入 V3 内部优化路线，统一进入 `docs/paper_reproduction/` 和 `external_baselines/`。
- V1 PINN、V2 历史路线、V3 当前主线必须分开记录；可以在总纲中横向总结，但不要在执行表里混写。
- 新实验结果优先进入路线日志或状态表；只有路线定位改变时才更新 `docs/实验设计总纲.md`。
- 点云回构、插值和 CFD-Post/Fluent 展示属于可视化交付链，不能替代同点定量评估。

## 3. 推进记录格式

`docs/02-推进与变更/代码修改与实验推进记录.md` 必须保持越新越靠上。新记录插入在文件说明块和第一条历史记录之间。

每条记录至少包含：

- 本次主要修改
- 对应代码/文档
- 推进到实验步骤
- 当前状态判断

建议标题格式：

```markdown
## YYYY-MM-DD｜主题 · 动作 · 状态
```

状态应写清楚 `已提交`、`训练中`、`TIMEOUT`、`完训`、`No-Go`、`待验证` 等，不使用“最近”“刚刚”等相对时间。

## 4. 清理检查清单

每次整理文档或收尾前，至少检查：

- `AGENTS.md` 是否只保留硬规则，没有变成历史流水账。
- `docs/README.md` 的入口是否能带新人找到总纲、任务 A、推进记录、外部 baseline 和汇报材料。
- 同一实验状态是否只在一个主文档中维护，其他位置只做链接或摘要。
- 已完成的交接文档是否归档到 `_archive/`，并在当前路线文档中保留最终结论。
- 进度记录是否位于文首，且包含四个必需部分。
- 文档中是否残留 `今天`、`昨天`、`最近`、`上周` 等相对时间。

## 5. 不应做的事

- 不因为整理文档而移动或删除 `data_new/`、`outputs/`、`stl_data/` 等大数据或证据目录。
- 不把外部 baseline 的训练状态写成 V3 内部消融。
- 不把失败路线删除；应写成历史证据、No-Go 原因或后续边界。
- 不在 `AGENTS.md` 里堆单次实验历史。
