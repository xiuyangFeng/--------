## 一、背景与触发

对正式主线层（作业 3537–3547，seed=1）结果进行深度分析时，发现：

1. **`-PW` 系列 best_epoch 极小（9–11 轮）**：Main-01-PW、WSS-01-a/b/c-PW 在 9–11 轮便触发 `is_best`，之后 30 轮 patience 全部空跑。训练并未"中途停止"（实际跑了 39–41 轮），但最优 checkpoint 保存过早。
2. **Anchor-01 best_epoch=5**：V1 路径同样在极早轮次锁定 best，后续 30 轮 val_wss_r2_wss 持续改善但 checkpoint 未被更新。
3. **WSS R² 整体在 0.33–0.40 之间**：与单目标探针（历史单次约 **0.397**、修复后重跑约 **0.364**，见 §10.6）同属该带宽，主线已接近经验天花板；仍存在 val/test 的显著差距（val 期间观测到 0.45–0.49，test 集多落在 0.33–0.40）。

通过逐 epoch 逐指标分析 `history.csv` 和 `summary.json`，最终定位到 **`trainer.py` 中 val_score 公式的 3 处设计缺陷**，以及配置层面的 2 处不合理设置。

## 二、OOD 残余困难的说明（回应疑问）

> **Q：PENG 已经被筛除了，为什么 WSS 精度还是这么低？**

### 2.1 OOD 记录澄清

- Diag-00（2026-05-03）的 OOD 警示（"WSS 测/训 q99 比率 41×"）是在 **PENG 移除前** 对原始 split 计算的。PENG 的 WSS 高达训练集均值的 275 倍，几乎单独拉高了整个测试集的分布统计。
- OOD 诊断文档 §3 已确认：**排除 PENG 后，剩余 17 个测试病例全部落在训练集分布范围内**（per-case z-score 均在 ±3σ 以内）。从数据分布角度，后续实验的测试集已不存在严重 OOD。

### 2.2 WSS 精度低的真实原因（与 OOD 无关）

| 原因 | 解释 |
| --- | --- |
| **任务固有上限低** | WSS 单目标探针在历史单次 run 上约 **0.397**、修复后重跑约 **0.364**，与 Anchor **~0.367** 同量级；多目标配方只会更低，整体落在 **~0.33–0.40** 带宽，可视为 WSS 在当前数据与表示下的**经验可学性上限区** |
| **分量不均衡** | wss_z 分量 R²≈0.35；wss_x/y 分量 R²≈0.01–0.07（接近不可学），均值被拉低 |
| **val 集样本少，单轮评估噪声大** | val 集仅 5–8 个 case，每轮 val_wss_r2_wss 在 0.33–0.49 大幅震荡（±0.08），test 集 17 个 case 统计更稳定但较低 |
| **val/test 系统差距** | val 集和 test 集的 case 组成不同，模型在 val 上碰到"好 case"时指标偏高；这是数据量不足的固有问题 |
| **最优 checkpoint 选取错误** | 由于 val_score 公式缺陷（见第三节），被保存的 best checkpoint 并不对应最好的 WSS 模型 |

**结论**：数据质量（OOD）问题已通过移除 PENG 得到解决；WSS 精度低的根源是任务本身的难度 + val_score 公式缺陷导致 checkpoint 选取不当，而非数据异常。

## 三、核心问题诊断

### 问题 1：`val_score` 公式中速度项为常数，主导 97% 的分数（最高优先级 Bug）

**位置**：`training/core/trainer.py`，第 306–322 行

**当前代码**：
```python
val_score = (
    wss_rmse / wss_std          # ~0.24（有效信号，范围 0.015）
    + wall_p_rmse / p_std       # ~0.000026（见问题 2）
    + 0.3 * vel_rmse / vel_std  # 对 PW 实验 = 常数 8.023（占 97%！）
)
```

**数值实测（Main-01-PW，41 个 epoch）**：

| 项 | 数值 | 说明 |
| --- | --- | --- |
| `wss_rmse / wss_std` | 0.225–0.240 | 有效但范围仅 0.015 |
| `wall_p_rmse / p_std` | 0.0000257 | 几乎为零（见问题 2） |
| `0.3 * vel_rmse / vel_std` | **8.023（固定不变）** | vel 未监督 → vel_rmse 恒为 1.4519；vel_std=0.054 极小 → 放大 26.8× |
| **val_score 总计** | 8.248–8.263 | 41 轮中变化范围仅 **0.015** |

**后果**：
- val_score 97% 是常数偏置，实际早停信号只剩 0.015 范围内的随机游走
- `best_epoch=11` 仅仅是第 11 轮 val_score 碰巧是 41 轮中的最低点（差值 0.0002）
- 第 34 轮 val_wss_r2=0.492（更优），val_score=8.2537（仅高出 0.0054），因 patience=30 已用尽而未被保存

**对含速度实验（Main-01）的影响**：速度被监督时，vel_rmse 会随训练减小（从 1.45 降至 0.3–0.5），vel 项不再是常数，早停信号有意义 → 故 Main-01 best_epoch=56 是正常的，不受此 bug 影响。

---

### 问题 2：`wall_p_rmse / p_std` 量纲混用，压力对早停贡献为零

**位置**：`training/core/trainer.py`，第 308、320 行

**根因**：

```python
p_std = self.norm_stds.get("p", 1.0)
# 从 normalization_params_global.json 读取的 p_std = 2824.4 Pa（物理单位）

wall_p_rmse = val_metrics.get("loss_wall_pressure", 0.0) ** 0.5
# loss_wall_pressure 是在归一化（z-score）空间计算的 MSE，sqrt 后 ≈ 0.07（无量纲）
```

两者量纲不同：分子是归一化空间的 RMSE（无量纲，O(0.1)），分母是物理单位 Pa（O(1000)）。

```
wall_p_rmse / p_std = 0.073 / 2824 ≈ 0.0000257 ≈ 0
```

**后果**：压力的改善/退化完全不影响 val_score，早停对压力质量完全不敏感。

**对比 WSS 项**：`wss_rmse / wss_std = 0.73 / 3.06 ≈ 0.24` —— WSS std 恰好与归一化 RMSE 量级接近，因此 WSS 项"碰巧"有效，但本质上量纲同样混用，只是 WSS 的物理 std（3 Pa）与归一化 RMSE（0.73）相差不大。

---

### 问题 3：`best_val_loss` 字段含义混淆（summary.json 可读性问题）

**位置**：`training/core/trainer.py`，第 439 行

```python
"best_val_loss": best_val,  # 实际存储的是 val_score（8.248），不是 val_loss（0.187）
```

`summary.json` 中 `best_val_loss=8.248` 与 `history.csv` 中 `val_loss=0.187` 完全不同，导致混淆。训练代码实际上追踪的是 `val_score`（复合 RMSE 指标），并不是 `val_loss`（训练 loss）。

---

### 问题 4：patience=30 对强噪声的 val_score 过短

在 val_score 范围仅 0.015 的情况下，patience=30 意味着只要有任意一轮随机最低点出现，之后 30 轮无论多好都无法"超过"它——因为差值精度太小，极容易在某个随机波动较低的早期 epoch 触发锁定。

当 val_score 范围扩大后（修复问题 1/2 后），patience=30 仍然偏短，推荐 50–80。

---

### 问题 5：warmup_epochs=0，初期 val 指标不稳定

所有 V3 实验均设 `warmup_epochs=0`，前几轮 LR 处于全速状态，模型参数剧烈更新，导致前 5 轮的 val_score 方差极大。若第 1–5 轮碰巧出现低谷（如 Anchor-01 epoch=5），会永久锁定 best，后续无法改善。

## 四、修复方案

### Fix-1：修复 val_score 公式（最高优先级）

**文件**：`training/core/trainer.py`

**修改位置**：第 306–322 行

**修改前**：
```python
if self._use_domain_loss and self.norm_stds:
    wss_std = self.norm_stds.get("wss", 1.0)
    p_std = self.norm_stds.get("p", 1.0)
    vel_std = max(
        (self.norm_stds.get("u", 1.0) + self.norm_stds.get("v", 1.0) + self.norm_stds.get("w", 1.0)) / 3.0,
        1e-10,
    )
    wss_rmse = val_metrics.get("wss_rmse")
    if wss_rmse is None:
        wss_rmse = val_metrics.get("loss_wall_wss", 0.0) ** 0.5
    wall_p_rmse = val_metrics.get("loss_wall_pressure", 0.0) ** 0.5 if "loss_wall_pressure" in val_metrics else val_metrics.get("rmse_p", 0.0)
    vel_rmse = val_metrics.get("rmse_vel_mag", 0.0)
    val_score = (
        wss_rmse / wss_std
        + wall_p_rmse / p_std
        + 0.3 * vel_rmse / vel_std
    )
```

**修改后**：
```python
if self._use_domain_loss and self.norm_stds:
    wss_std = self.norm_stds.get("wss", 1.0)
    vel_std = max(
        (self.norm_stds.get("u", 1.0) + self.norm_stds.get("v", 1.0) + self.norm_stds.get("w", 1.0)) / 3.0,
        1e-10,
    )
    # WSS RMSE（归一化空间），除以 wss_std（物理单位 Pa）后 ~0.2 量级
    wss_rmse = val_metrics.get("wss_rmse")
    if wss_rmse is None:
        wss_rmse = val_metrics.get("loss_wall_wss", 0.0) ** 0.5

    # [Fix-1a] 压力 RMSE 已在归一化空间（无量纲），直接使用，不除以物理单位 p_std
    # 乘以系数 3.0 使其量级与 wss_rmse/wss_std 相近（~0.07*3=0.21 vs ~0.24）
    wall_p_rmse = (
        val_metrics.get("loss_wall_pressure", 0.0) ** 0.5
        if "loss_wall_pressure" in val_metrics
        else val_metrics.get("rmse_p", 0.0)
    )
    _P_SCALE = 3.0  # 使压力项与 WSS 项量级匹配

    # [Fix-1b] 仅当速度被实际监督时才纳入速度 RMSE 项
    # 对 PW 实验（lambda_vel_int=0），vel_rmse 为常数（模型未学习速度），
    # 不应纳入 val_score，否则 97% 的分数被常数占据，早停失效
    _vel_supervised = (
        self.domain_loss_config is not None
        and getattr(self.domain_loss_config, "lambda_vel_int", 0.0) > 0
    )
    if _vel_supervised:
        vel_rmse = val_metrics.get("rmse_vel_mag", 0.0)
        vel_term = 0.3 * vel_rmse / vel_std
    else:
        vel_term = 0.0

    val_score = wss_rmse / wss_std + _P_SCALE * wall_p_rmse + vel_term
```

**修复后预期数值（Main-01-PW）**：

| 项 | 修复前 | 修复后 |
| --- | --- | --- |
| WSS 项 | 0.225–0.240 | 0.225–0.240（不变） |
| 压力项 | ~0.000026（≈0） | ~0.21（有意义） |
| 速度项 | 8.023（常数） | **0.0（删除）** |
| val_score 总计 | 8.248–8.263（范围 0.015） | **~0.43–0.45（范围 ~0.02，信噪比大幅改善）** |

**对含速度实验（Main-01）的影响**：`_vel_supervised=True`，速度项保留，行为与修复前基本一致（vel_rmse 减小时分数下降，早停信号有效），best_epoch 不会受影响。

### Fix-3：增大 patience 并加入 early_stop_min_delta

**文件**：所有 V3 生成配置（`training/configs/field/generated/v3_pointcloud/*.json`）

**修改方向**：
- `early_stopping_patience`: 30 → **60**（对噪声信号更宽容）
- 在 `trainer.py` 中加入 `early_stop_min_delta`：val_score 只有改善超过 `min_delta` 才算刷新 best

**代码修改（trainer.py `__init__`）**：
```python
self.early_stop_min_delta = float(getattr(optim_config, "early_stop_min_delta", 1e-4))
```

**代码修改（trainer.py `fit()` 中 is_best 判断）**：
```python
# 修改前
is_best = val_score < best_val

# 修改后
is_best = val_score < (best_val - self.early_stop_min_delta)
```

### Fix-4：加入 warmup_epochs（建议值 5）

**文件**：V3 生成配置

前 5 轮 LR 从 0 线性爬升，避免初期剧烈参数更新导致 val_score 波动过大、早期锁定错误 best checkpoint。

**配置修改**：
```json
"optim": {
    "warmup_epochs": 5,
    "early_stopping_patience": 60
}
```

> 同时在 `trainer.py` 中将 warmup 阶段（epoch ≤ warmup_epochs）的 epoch **不参与** early stopping 判断。

## 五、单轮评估噪声降低策略

当前问题：val 集仅 5–8 个 case，每轮评估的 val_wss_r2_wss 在 0.33–0.49 之间大幅震荡（σ ≈ 0.04）。

### 策略 A：EMA 平滑 val_score（推荐与 Fix-1 配合使用）

```python
# trainer.py __init__:
self._ema_alpha = 0.4       # 平滑系数（越小越平滑，越慢响应）
self._ema_val_score = None  # EMA 状态

# fit() 中，每轮计算 val_score 后：
if self._ema_val_score is None:
    self._ema_val_score = val_score
else:
    self._ema_val_score = self._ema_alpha * val_score + (1 - self._ema_alpha) * self._ema_val_score

# 用 EMA 值做 early stopping
is_best = self._ema_val_score < (best_val - self.early_stop_min_delta)
```

- 3 轮 EMA（α=0.4）等效于约 2.5 轮的加权平均，大幅减少单轮噪声
- 不影响学习率调度器（继续用原始 val_score 驱动 ReduceLROnPlateau）
- 代价：best_epoch 判定会有约 2–3 轮的滞后，可接受

### 策略 B：增大 patience（已在 Fix-3 中涵盖）

patience=30 → 60，给模型足够时间从噪声最低点"逃逸"。

### 策略 C：多重 checkpoint 保存（Fix-5 已涵盖）

同时保存 best_val_score 和 best_val_wss_r2 两个 checkpoint，测试时同时评估两个，取 WSS 更好的作为最终结果。

### 策略 D：增加 val 集大小（数据层面，长期）

当前 val 集 5–8 例，建议调整 split 使 val 达到 10–12 例，以降低每轮评估的统计噪声。此策略需要重新划分 split 并重跑所有实验，建议下一阶段实施。
