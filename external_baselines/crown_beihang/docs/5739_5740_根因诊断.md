## Job 5739：非 PINN 为何 OOM

结论：5739 的 `FAILED` 首因是工程 OOM，不是训练循环本身先崩。`external_baselines/crown_beihang/cluster/run_crown.slurm` 配置 `--mem=32G`，而旧 `build_datasets(config)` 在训练启动时无条件读取 train/val/test 三个 merged pkl。仅 `crown_volume_train.pkl` 就是 100G，再叠加 val/test、pickle 对象开销、DataLoader batch 中全点云张量、GPU/CPU 中间拷贝，32G 必然不足。日志 `cluster/logs/crown_5739.err` 显示 epoch 4 进程被 `Killed`，符合 cgroup/系统 OOM 特征。

非 PINN 的模型质量结论需要单独看：5739 虽在 OOM 前保存了 `best_epoch=2`，并已离线补评 test，但 test `p_r2=-4.3677`、速度 R2 也未达可接受水平，已在 `experiments/crown_original_vp/实验分析记录.md` 和总账文首记录为 No-Go。也就是说：

- 工程状态：旧加载路径导致 OOM，run 未正常完成。
- 模型状态：即使用 best checkpoint 补评，当前非 PINN raw_ascii v1 仍 No-Go。

lazy 修复后，`build_datasets(config)` 默认返回 `CrownLazyDataset`，只建索引，不再常驻 100G train merged pkl。实测构建三 split 并触碰 4 个 partial shard的审计进程 Max RSS 约 7.5GB；完整 lazy test evaluate Max RSS 约 12.3GB。训练仍建议把 Slurm 内存从 32G 上调到至少 64G，因为 `batch_size=16` 时 batch 内会同时持有多帧全点云 tensor，`pin_memory=true` 还会增加 pinned CPU 内存；如果后续开 `num_workers>0`，每个 worker 会有独立 dataset/cache，内存和 IO 放大更明显。

推荐非 PINN 重训参数：

| 项 | 建议 |
| --- | --- |
| `--mem` | 64G 起步；若开 workers 或提高 batch，建议 96G |
| `--time` | 48:00:00 起步，避免再次在未收敛前被墙钟截断 |
| `batch_size` | 先保持 16；若仍 OOM，降到 8 |
| `num_workers` | 保持 0；确认 IO/内存后再试 1-2 |
| `partial_cache_cases` | 随机 shuffle 下 2 能控内存但命中低；若内存给到 96G，可试 4-8 改善 IO |

## Job 5740：PINN 为何极慢

结论：5740 的 `TIMEOUT` 是 PINN 计算路径和旧/低命中数据读取共同造成的长 epoch。`history.csv` 只有 23 行，`cluster/logs/crown_5740.err` 显示 2026-06-19 14:51:43 因 24h time limit 取消；best checkpoint 来自 `epoch=20`，best val loss 为 `0.0099044435`。

step 数量本身已经很大：

| 阶段 | 样本 | batch_size | steps/epoch |
| --- | ---: | ---: | ---: |
| train | 4617 | 2 | 2309 |
| val | 648 | 2 | 324 |
| 合计 | - | - | 2633 |

24h 仅完成约 23 epoch，折合约 1.0 小时/epoch、约 1.3 秒/step。这个量级与代码路径一致：

- `_prepare_minibatch(..., separate_phy=True)` 对每个样本生成两套随机索引：data idx 和 physics idx。
- `train_step_pinn()` 每 step 至少三次 forward：data forward、wall loss 的壁面 forward、`physics_residual_loss()` forward。
- `physics_residual_loss()` 对 `u/v/w/p` 计算一阶和二阶坐标导数，包含多次 `torch.autograd.grad(..., retain_graph=True, create_graph=True)`，图保留和二阶导显著放大显存、算力和调度成本。
- `eval_step_pinn()` 在验证阶段也必须 `torch.enable_grad()`，因为 PDE residual 需要梯度；所以 PINN val 不是普通 no-grad evaluate。
- lazy 前训练若反复读 partial/merged 大对象，数据侧会进一步拖慢。即使 lazy 后，train loader `shuffle=True` 是按帧随机，57 个 train case 随机交错；`partial_cache_cases=2` 的命中率很低，容易频繁加载 1-2GB 级 partial shard，形成 IO thrashing。

`partial_cache_cases=2` 的优点是控制内存，缺点是对随机帧级 shuffle 不友好。val/test 的索引按病例连续，cache 命中较好；train 的随机顺序会让 cache 基本只承担“限制峰值”的作用，而非显著加速。

## 可落地加速建议

不改论文口径的前提下，优先做工程调参：

| 项 | 建议 |
| --- | --- |
| PINN evaluate | 使用专用 Slurm，只构建目标 split；正式指标从集群作业产物入账 |
| `--mem` | PINN 训练/evaluate 均建议 64G 起步；若 `partial_cache_cases>=4`，建议 96G |
| `--time` | PINN 训练建议至少 72:00:00；若目标是完整 early-stopping/稳定收敛，建议 96:00:00 |
| `partial_cache_cases` | 训练可试 4 或 8；若 RSS 仍可控，能降低随机 case thrashing |
| `num_workers` | 默认 0 最稳；开 worker 会复制 dataset/cache，除非明确验证内存和 IO，否则不建议 |
| `pin_memory` | GPU 训练可保留；若 CPU 内存压力明显，可临时关掉比较 |
| evaluate-only | 保留 `cluster/run_crown_evaluate.slurm`，避免训练作业 timeout 后无法正式补评 |

如果允许后续做非口径改变的优化，可以考虑 case-aware batch sampler：按 case 成组取样以提高 partial cache 命中，再在 epoch 内打乱 case 顺序。但这会改变 batch 采样相关性，建议作为单独工程优化记录，不混入当前论文复现实验口径。
